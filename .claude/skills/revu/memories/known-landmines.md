# Known landmines — traps already identified

Each of these has a ticket or a doc section; listed here so no session rediscovers them the hard way.

**In the prototype (fix before real GitHub traffic — milestone M1):**
- `looksLikePersonName` in `src/lib/identity.ts` admits only letter-tokens, but Coder usernames can contain digits/underscores (`alice2`, `j_doe`) — such a contractor's every comment would render as the bare bot. Relax the charset + add an `alice2` fixture (M1.2) **before the first stamped comment**.
- `BROKER_LOGIN` is a hardcoded const in `src/api/types.ts` — but `session.brokerLogin` already exists in the type and is populated by the mock; the bug is that `lib/identity.ts` imports the const instead of reading the session (M1.1).
- Own-comment detection is name-string matching — breaks across Coder renames/username reuse; upgrade to the `commentAuthors` id-map with name fallback (M1.3).
- Split-mode preference sits in `sessionStorage` (`src/pages/files.tsx`) — the last browser-side per-human state; fold into the prefs store (M1.4) to make "rebuild loses nothing" literally total.

**GitHub API:**
- GraphQL thread comments must be normalized to the REST `ReviewComment` vocabulary — request `fullDatabaseId` for REST-numeric ids, map `diffSide`→`side`, carry `diffHunk`→`diff_hunk`. **Confirm `fullDatabaseId` still exists in the current schema** before building on it (M2.4).
- `replyToThread`: REST wants a *comment* id, the contract passes a *thread* id — reply to the thread's **first** comment; GitHub attaches replies to the root regardless.
- Reactions are per-GitHub-user and there is one bot user: everyone's 👍 is the same 👍. Ship shared-and-honest; **do not** build per-human reaction simulation.
- Installation tokens *can* run `resolveReviewThread`/`unresolveReviewThread` GraphQL mutations; `resolvedBy` reads as the bot (UI already renders that).
- Blobs: local `git cat-file` first (zero API cost, works offline); API fallback batched via GraphQL `object()` aliases ~30/query. Binary heuristic = NUL byte in first 8000 bytes, same as git.
- **A base-branch advance moves the three-dot compare but does NOT refresh the pull's stored `base.sha` until a "synchronize"** (a head push, or a `PATCH pulls/{n}` base edit / `gh pr edit --base <same branch>`). The direct sync reads `pull.base.sha` to compute `merge_base…head`, so a base advance with no head push is invisible until GitHub recomputes `base.sha` — relevant to M4's live-layer base-change detection, and the reason the base-advances live check (`scripts/smoke-direct.ts`) nudges the base with a re-edit before re-syncing.
- **A GitHub App installation token CANNOT call `GET /user`** — GitHub answers 403 "Resource not accessible by integration" (that endpoint needs a user-to-server or personal token; an installation token acts *as the app*, which has no user). So broker mode (which authenticates with the injected installation token) must NOT probe the viewer at boot — `viewerLogin` is left **absent** (the shared-bot session shape), and boot resolves identity locally from git config only. Probing `GET /user` crash-loops broker boot exactly when the credential IS present (the steady state) — the adversarial review of M3.8-core caught this.

**Test harness (`bun test`) — bit CI twice during M0:**
- **One shared mock store across all files.** The `test/preload.ts` `localStorage` shim is a single process-wide `Map`, and the mock store persists to it (debounced ~1s). `bun test` re-imports the store per file but the shim is shared, so one file's mock mutations — or a debounced flush of them — leak into another file's `load()`. Green locally, red on the slower CI runner (order/timing dependent). **Any test that drives the mock (`createMockApi`/`mockDev`) must `mockDev.reset()` in `beforeAll` (and `afterAll` if it mutates).** The scenario walk assumes a pristine fixture seed.
- **The gate runs `bun test` BEFORE the app build** (`oxlint && tsc -b && bun test && build`), so `packages/app/dist` does not exist during tests. Any test that starts `revud` must point it at a stub dist via `REVU_DIST_DIR` (+ a temp `REVU_DATA_DIR`), never the real build.

**Transport modes (M0+):**
- The app selects `createMockApi()` unless `VITE_REVU_API` is set at build time; `?mock=1` ALWAYS forces the pure in-browser mock (no HTTP). `dev:e2e` builds with `VITE_REVU_API=/` so the served app makes same-origin relative `/api/*` calls to revud (port-independent). A no-arg `bun run build:app` produces a mock build.
- `addReaction` carries the owning PR as `?pr=<n>` — the contract route path (`/api/comments/:id/reactions`) has only the comment id, but the mock/revud need the PR to locate the comment.

**E2E (playwright-core, headless system Chrome) — the browser gate (MT.4+):**
- The e2e driver is a **plain `bun run` script, never a `*.test.ts`** (`e2e/happy-path.ts` + `e2e/harness.ts`, run via `bun run test:e2e`). The gate runs `bun test` BEFORE the app build, so a `*.test.ts` e2e would be swept into the gate and run with no `dist` present. It does its own `check()` assertions and `process.exit(1)` on failure, exactly like `scripts/smoke.ts`. CI runs it as a **separate `e2e` job** with `needs: check` (after the gate), capped `timeout-minutes: 10`.
- The e2e needs the **HTTP-mode** dist (`build:e2e` = `VITE_REVU_API=/ bun run build:app`); a no-arg `build:app` is a mock build and would silently pass the whole UI flow without touching the network. The happy path asserts **≥1 real `/api/*` request** fired (the seam is genuinely exercised) and that `?mock=1` fires **zero** — that pair is the only guard against an accidental mock build.
- **Happy-path fixture = PR 101:** never-synced, no seeded draft, authored by another human with `h-priya` (the default human) as reviewer, so it appears in the inbox. Clean `sync → inline draft comment → COMMENT submit → draft cleared`. Avoid 312 (seeded draft) and 389 (reconcile).
- Two selector traps that were green-locally/red-on-a-cold-runner: (1) a non-auto-waiting `isVisible()` probe on the `Sync now` button races first render — **wait for the button, don't probe**; (2) `getByText('synced')` also matches "This PR was never synced" / the "never synced" seal — match the success seal `⧗ <sha> · synced <time>` with **`/· synced/`** instead.
- Chrome on CI: `channel: 'chrome'` resolves the preinstalled Google Chrome on `ubuntu-latest` (confirmed: e2e job green in ~14s); `E2E_CHROME_PATH` overrides `executablePath` as an escape hatch. No browser-download step. The harness serves the real dist via a fresh temp `REVU_DATA_DIR` + ephemeral `REVU_PORT=0` per run (pristine fixtures), so the shared-mock-store `mockDev.reset()` rule does not apply here.

**Linear MCP (tracking):**
- Milestone names ≤ 80 chars or `save_milestone` fails. Milestone comments need the milestone **UUID** (`list_milestones` first). Status-update reads need `orderBy:"createdAt"`. `list_issues` has no milestone filter. Real newlines in bodies, never literal `\n`.

**CHECKPOINT_1 (independent pre-M2 review of `main` @ `5985f34` → milestone M1.6) — traps + do-not-undo:**
- **`/api/dev`, `/api/dev/reset` are mock-only forever (C3).** They change identity (`dev.setHuman`) and reseed the store; gated at the revud router boundary on an **explicitly-passed** mode (not `process.env`, so a stray env var can't re-enable it). **M3/direct must never expose them.** They sit outside the shared `ROUTES` table by design — documented in `http.ts`.
- **Durability is a daemon concern, not a browser one (C5).** The mock's `flush()`/`load()` swallow storage errors — correct for localStorage, wrong on disk. revud surfaces write failures as typed errors (never 200 on an unpersisted draft) and distinguishes absent from unreadable (never reseed over a present-but-unreadable document). **M2.3's SQLite store owns durability end-to-end: no error swallowing on the write path; unreadable ≠ absent.** Do not inherit the browser swallow into the durable path.
- **`STORE_VERSION` migrations migrate in place, never reseed (C4 — restates the M1.4 P0).** Any store-shape change bumps the version and defaults new fields in `load()`'s migration ladder; a bump that reseeds wipes drafts and violates "drafts survive everything." A new field (e.g. `startLineText`) defaults to null on old documents.
- **`anchor.ts` reconcile logic is shared truth, written once (C1/C2).** Blob-side selection (base for LEFT, head for RIGHT) lives in ONE selector imported by both the adapter and the reconcile dialog; the `clean` fast path requires a context-score floor (a coincidental duplicate line at the original index must not short-circuit to `clean`). The conformance suite is where reconcile correctness is proven — add scenarios (LEFT-side, preview/report parity), don't re-verify by hand.

**Do NOT "fix" these — correct unasked-for calls a checkpoint could provoke undoing (CHECKPOINT_1 §What-is-solid):**
- **Reusing the app's mock in revud** (`import('@revu/app/mock')` + a disk-backed `Storage` polyfill) instead of M0.3's specified port — one oracle, no divergence. Keep it.
- **The `load()` migration ladder + the M1.4 in-place prefs migration** (a version bump never wipes drafts) — exactly right.
- **The `network`-code enveloping** (client-side-only code, no HTTP status, surfaced as an enveloped 5xx and reconstructed adapter-side) — a careful, correctly-documented call at both ends.
- **DOM-global guards** (`typeof document`/`typeof localStorage`) for headless reuse, and the boot-order dependency (install storage → then load mock) — load-bearing, documented, tested.
- **MT** (the TDD milestone inserted ahead of M0) and **the transport-parameterized conformance suite** — the right assets; C1/C2 are gaps in its *scenarios*, not its design.

**M2 (direct mode — first real GitHub) — decisions:**
- **Direct-mode auth = the `gh` user, NOT the GitHub App.** The direct `TokenSource` resolves the token via `gh auth token` (respecting `GH_TOKEN`/`GITHUB_TOKEN`); owner/repo from `git remote get-url origin` (guide §5 + MILESTONES.md M2.2). The App private key `tmp/revu-sandbox-app.pem` + installation-token minting is **M3's separate `TokenSource` strategy** (broker mode), not M2. A prior handover's "run as the bot in M2" was aspirational — the doc is source of truth. **Do NOT wire the `.pem`/App into M2 direct mode.** Non-secret App coords (App id 4326148, installation 147260896) live in `tmp/appdetails.md` and are for M3.
- **Mode = an injected pair of strategies (`TokenSource`, `WriteDecorator`) around one shared core** (guide §5). The core — sync engine, snapshot store, draft store, anchor — is ~all the code; mock/direct/broker differ only in the injected pair. M2.2 builds `TokenSource`; M2.6 builds `WriteDecorator`.
- **Scratch repo `pat-mw/revu-sandbox` is seeded as the gh user** by `scripts/seed-scratch.ts` — idempotent (re-run converges, no duplicate PRs), guarded to the scratch target so it can never mutate a real repo. Wired into conformance CI (M5.1). The detailed sync/normalize/write/blob traps for M2.3–M2.7 already live in the "GitHub API" + "Transport modes" sections above and in `docs/agent/INTEGRATION_GUIDE.md` §3–§4.

**CHECKPOINT_2 (M3 broker plan validated against the real sandbox → `docs/agent/CHECKPOINT_2.md`) — decisions + do-not-undo:**
- **Deployment is MINT-AND-INJECT, NOT proxy (topology correction).** The host token broker PUSHES a ~1h repo-scoped installation token into each container's `~/.git-credentials` over `docker exec` stdin; there is NO listening socket and no inbound service a workspace can call. The workspace's own `git`/`gh` call GitHub DIRECTLY as the App. So "broker mode" is the M2 engine running IN the workspace on the ambient injected token — reads + writes + reconcile — via a **`FileCredentialTokenSource`** (parse `https://x-access-token:<tok>@github.com`, re-read per request, tolerate ~40min rotation + 60s cold gap + the 401-erase-of-the-file). **There is NO GitHub proxy and NO workspace-callable revu listener** — the guide-§2 proxy model + the tailnet broker (`REVU_BROKER_URL=broker.tail…`, tailnet ACL) are the exact inbound service the owner refused; do not build them. Custody claim is "revu adds no new credential and never serializes tokens", NOT "the workspace never holds a token" (false here — the workspace holds a live GitHub write token regardless).
- **Audit = provenance + out-of-band detection, NOT prevention (no permission split).** The injected token keeps `pull_requests:write`, so a contractor can `curl` `api.github.com` and post bot-identity comments directly, bypassing revu. revu's audit is authoritative provenance of writes it mediates + a **detector**: the host reconciler flags any App-authored comment/review id absent from every journal as an out-of-band write. The owner chose NOT to split permissions (contractors keep direct `gh`) — so audit is detection, not prevention. Say so honestly in M5.4.
- **Identity binding = `coder.owner` via the host tick, NEVER workspace-claimed.** The only host-authenticated identity is the Coder username (`coder.owner` container label, set by the Apoha Terraform template, zero contractor-editable params). Audit identity + the durable-store email key derive host-side from `coder.owner`→email (map / Coder API), never from git config, headers, or anything revud self-reports (all spoofable by a contractor with sudo). Draft access is authorized by the `coder.owner` binding, NOT by a path `:email` param (the specced `/v1/drafts/:email/:n` would let any workspace read any human's drafts). M3.6 is push-only: there are NO workspace→broker calls to authenticate — drop tailnet-source + mTLS (workspaces are not tailnet nodes; Lima/Docker NAT collapses all workspaces to one source address; mTLS has no provisioning path).
- **The host-side "revu collector" is the sandbox ADAPTER; revu core stays generic.** A thin host component beside the token broker rides the existing 60s `docker exec` tick to PULL each container's drafts + audit journal (`coder.owner`-bound), holds the durable store, and runs the out-of-band detector. No listening socket. Keep the sandbox-shaped pieces (collector, `coder.owner` binding, `FileCredentialTokenSource`) behind the adapter seam — the published direct-mode tool must not inherit them (M7).
- **App grant = `contents:write` + `pull_requests:write` + `metadata:read` + `checks:read`.** The real App originally lacked `checks:read` (guide §3 step 5 `check-runs` would 403 on-prem); the owner ADDS `checks:read`. Scratch App must be **byte-identical** to this grant so conformance fails early; NO webhooks (poll-only is correct). Add a per-RevuApi-method → App-permission matrix (M3.1).
- **revud binds `127.0.0.1`, serves one port via `coder port-forward --tcp`, baked at `/opt/revu`.** Loopback removes revu from the cross-container threat class on the shared docker bridge (the port-forward agent is co-resident). Named `coder_app` path-apps are broken for SPAs (no wildcard TLS); port 3000 collides with Coder. Home-volume seeding happens once and never propagates updates → bake at the system path `/opt/revu`; no revu state under `/home/coder` is authoritative.
- **Owner-actioned, NOT revu tickets (cross-boundary dependencies):** at-rest/backup/FileVault for the audit log = ops layer; the org ruleset stale-approval settings stay OFF (revu surfaces stale approvals honestly — residual: approve-then-push-then-merge is outside revu's control); container-to-container network policy deferred (loopback bind is sufficient for revu — residual recorded). Offboarding hook: retain audit rows, purge drafts/viewed.

**M3 (Broker mode — in-workspace inject) — build decisions + landmines (foundation landed: PRs #31 M3.2, #32 M3.8-core):**
- **Broker mode reuses the M2 direct engine wholesale** — the only injected difference is the `TokenSource`. `RevuMode` already included `'broker'`; `resolveMode` gained the branch; `resolveDirectContext` gained optional `tokenSource`/`validateToken` (direct defaults unchanged — mock/direct are byte-for-byte regression-free). `createStampingWriteDecorator(human, recordWrite)` already exists from M2.6 and stamps the body with the git-config **human** (not the bot) — broker writes reuse it.
- **`FileCredentialTokenSource` (`packages/revud/src/broker/token-source.ts`)** parses the `x-access-token` github.com entry from `~/.git-credentials`, **re-read per request** (never cached — the host rotates ~40min and 401-erases the file). Missing/empty/unparseable → a typed transient **`AwaitingCredentialError`** (distinct from the fatal `NoTokenError`), mapped at the router to `broker_unreachable` (502). Custody claim = "revu adds no new credential and never serializes tokens" (NOT "never holds a token" — false; the workspace holds a live write token). No-leak: read failures surface at most a sanitized errno mnemonic (`/^[A-Z][A-Z0-9_]{0,31}$/`), never reader-controlled content. WHATWG `URL.password` does NOT percent-decode, so `decodeURIComponent(url.password)` is exactly one decode (matches git's one-encode-on-write) — verified, no double-decode.
- **Broker boot must tolerate an absent credential** (the ~60s cold-start gap) — `buildBrokerSession` makes NO GitHub call at boot (see the `GET /user` 403 landmine above). Boot never hard-fails on a missing credential; the awaiting state surfaces per request.
- **Broker binds `127.0.0.1`** (loopback — the `coder port-forward` agent is co-resident; keeps revu off the shared docker bridge). mock/direct bind unchanged. `Bun.serve` `hostname` threaded through `startServer`. Note for M6.3: `127.0.0.1` is IPv4-only; a forwarder dialing `localhost`→`::1` would miss it.
- **Broker WRITES are one interlocking system — do NOT half-enable them.** An absent `viewerLogin` makes `canApprove` always-false AND breaks the submit idempotency-by-self re-check (`reviewMatches(r, viewerLogin='', …)` never matches → **double-post on a lost-response retry**); and a passthrough (unstamped) write violates M3 exit criterion 2. So M3.8-core gates the 4 write endpoints (submit/reply/resolve/react) to `not_implemented` (501) **in broker mode only, at the router** (mirrors how `listPulls` returns not_implemented; direct engine write path untouched). M3.4 turns them on as one unit: stamp-with-human + the App **bot login** (`<app-slug>[bot]`, sourced from config — an installation token can't self-discover it) for the submit self-guards/idempotency + the append-only audit journal (SQLite, GitHub-assigned ids, STORE_VERSION migrate-in-place) + the out-of-band-write detector.
- **In-gate broker conformance** lives beside direct's (`packages/revud/src/direct/conformance-broker.test.ts`, shared fakes in `conformance-fakes.ts`); it wraps each fake GitHub client with a `tokenGated` shim that calls `getToken()` before every method (mirrors the real client building the Bearer per request), a temp cred file + temp SQLite. Neither direct nor broker runs the full `runConformanceSuite` (the router answers `listPulls` with not_implemented) — both use a hand-written engine-subset, so broker isn't held to a weaker bar than direct.

**M3.4 (broker writes) — the interlocking system + its landmines (PR #34):**
- **`viewerLogin` serves TWO conflicting purposes in broker mode — get this wrong and you leak cross-human identity.** For the WRITE self-guards (the approve gate `authorLogin !== viewerLogin` + submit idempotency-by-self `reviewMatches(r, viewerLogin, …)`), broker must set `viewerLogin = the App bot login` (`<slug>[bot]`) so a write finds the bot's own prior review (no double-post) and can't self-approve. BUT for READ-side own-comment detection, `isOwnComment` signal 2 is `comment.user.login === viewerLogin` — and since ALL broker comments are authored by the one bot, `viewerLogin=bot` makes EVERY human's stamped comment read as "yours" (cross-human misattribution — the exact failure the product exists to prevent). Fix: signal 2 must skip bot-authored comments (`comment.user.login !== ctx.botLogin`) so they fall through to the stamped-name / write-log path. Broker snapshots never carry `commentAuthors` (M4), so the stamped-name path is the real own-comment signal there. Direct mode is unaffected (its `botLogin=''`).
- **In broker, `brokerLogin` AND `viewerLogin` both = the bot login** — this INVERTS direct mode's "brokerLogin must not echo viewerLogin" rule (in direct, viewer=human and echoing would misrender the human's own comments as the bot). In broker every write is the one bot and the human is in the stamped prefix, so bot-authored comments SHOULD route to the prefix parser. Different modes, opposite invariants — don't "fix" one to match the other.
- **Broker writes REQUIRE the bot login (`REVU_BOT_LOGIN`); unset → writes stay 501 (reads-only).** Without a self-identity the approve gate + idempotency are unsafe (double-post on retry). The bot login is deployment config — an installation token can't self-discover it (`GET /user` 403s); resolve it via `GET /app` (slug → `<slug>[bot]`), which `scripts/broker-mint-token.ts` prints. A MISCONFIGURED bot login silently defeats both guards, so the write path warns (token-free) when a write's real author login ≠ the configured one.
- **The write-gate is STRUCTURALLY coupled to the stamping decorator** (`api.brokerWritesEnabled`, true only for `createBrokerWriteDecorator`): the router gates on the api capability, NOT on the session, so a `viewerLogin`-bearing session over a passthrough api is reads-only (fail closed — never serve unstamped/unjournaled writes as the bot).
- **Audit journal** = append-only `audit_log` SQLite table (`github_id, human_id, workspace, endpoint, pr, created_at`), STORE_VERSION bump with **migrate-in-place** (drafts survive), durable `appendAudit` (surfaces `StoreWriteError`, never silent) + ordered `listAudit`. `recordWrite(githubId, {endpoint, pr})` appends after a confirmed write; the human_id is the git-config email (audit keys on email — fine, it's local/host, never in a GitHub body). The **out-of-band-write detector** (reconcile App-authored ids on GitHub vs the journal) rides the host collector (M3.7), NOT the in-workspace daemon — audit is detection, not prevention (the token keeps `pull_requests:write`; contractors keep direct `gh`).

**M3 host-collector infrastructure (M3.6/M3.5/M3.7.2/M3.9/M3.7.3 — in-gate; the docker-exec pull tick + launchd + `/opt/revu` bake are M6 hardware). New code under `packages/revud/src/collector/`:**
- **The one authenticated identity is `coder.owner`; EVERYTHING in a pulled payload is workspace-claimed and spoofable (contractor has sudo).** The `humanId` embedded in a pulled `ReviewDraft` and the `human_id`/`workspace` of a pulled audit row are display-only. The host store **re-keys every landed record to the `coder.owner`→email binding**, discarding the payload identity: `landDraft` overwrites `draft.humanId`; `landAudit` builds each row with `human_id = binding.email` + `workspace = coderOwner`, keeping ONLY `githubId/endpoint/pr/createdAt`. The collector passes `container.coderOwner` (channel-authentic) to the store and never reads identity from the payload. Git-config identity stays display-only (M3.6 trust-boundary doc, CHECKPOINT_2 §B).
- **The host store (`collector/host-store.ts`) is a DEDICATED sibling SQLite (`host.sqlite`), NOT `openDirectStore`.** It needs three things direct's store deliberately lacks: a `UNIQUE(github_id,human_id,workspace,endpoint,pr,created_at)` tuple for **idempotent audit landing** (direct's `audit_log` has no unique on purpose — a retry may legitimately double-journal), a **binding-keyed** API (`landX(coderOwner,…)` → `mustResolve` → email; no `:email` path exists), and **purge-by-human**. Reuses direct's durability discipline (WAL+FULL, `StoreWriteError`/`StoreUnreadableError`, absent-vs-unreadable, migrate-in-place) + `AuditEntry` type.
- **Idempotent landing = `INSERT … ON CONFLICT(<full tuple>) DO NOTHING`, NEVER `INSERT OR IGNORE`.** `INSERT OR IGNORE` swallows ALL constraint violations (incl. NOT NULL) — an attacker-controllable `githubId: null` would silently vanish from the permanent journal. `landAudit` also **validates each pulled row host-side** (`githubId`/`pr` positive safe ints; `endpoint`/`createdAt` non-empty strings, length-capped) and returns a `rejected` list (no silent swallow, no poison-pill — one bad row never blocks the human's good rows).
- **The out-of-band detector runs over the MERGED UNION (`listAuditUnion`), per-PR, AFTER all containers land — NEVER per-container.** One shared bot authors for everyone, so a write mediated by container A is absolved only when reconciling against the union of all journals; per-container detection would false-flag every other human's mediated writes. Land-all-then-detect ordering is load-bearing.
- **The collector tick (`collector/collector.ts` `runCollectorTick`) treats the pull result as untrusted `unknown`** — every structural layer is runtime-checked (a malformed pulled element must NOT throw out of the tick and deny detection for ALL PRs). Failure-isolated: unbound owner → `bound:false`, nothing landed; other errors → sanitized `err.name` (charset-guarded, never message content); per-PR reconcile errors → a `reconcileErrors` list. **Detector PR-set completeness is a caller obligation:** activity-derived PRs are a SAFETY NET; the real host tick MUST pass the COMPLETE open-PR set via `extraPrNumbers` (a bypass on a PR with no pulled activity is otherwise unchecked). A hint-count cap is an M6 follow-up (GitHub-read amplification).
- **Identity binding (`collector/identity-binding.ts`) fails FAST on config that would silently mis-key:** duplicate-email across two owners (a silent identity MERGE onto one store/audit key) and case-collision (`alice`/`Alice`, one is dead config that drops a human) both throw at construction. The duplicate-email error names ONLY the two owner keys, **never the colliding email** — it reaches operator logs via the host-CLI config loader (`collector/config.ts`), and the keys alone locate the fix.
- **Offboarding (`collector/offboard.ts`): retain audit, purge working state.** `offboardHuman` only calls the store's transactional `purgeWorkingState` (drafts+viewed) + read-only `listAuditForOwner` — no path can delete an audit row. **Ordering: run it BEFORE the `repos.map` entry is removed** — purge/retain are keyed by the binding; after removal → `UnboundOwnerError` (nothing purged). Editing an owner's mapped email mid-engagement strands rows under the old email.
- **Audit export (`collector/audit-export.ts`, `scripts/broker-audit.ts`): faithful + hostile-field-safe.** Never drop/reorder/dedup rows. **CSV formula injection:** `endpoint` is workspace-authored (store bounds length, not charset), so a `=HYPERLINK(…)` cell must be neutralized — prefix a `'` when a cell starts with a formula trigger (`= + - @`) OR leading whitespace (an importer may trim it to expose the trigger), then quote-wrap. Table: replace control chars with a visible `?` (a hostile field can't break layout or emit terminal escapes). **`--since` MUST canonicalize to the store's ms-precision `.sssZ`** (`canonicalizeSinceBound`) — stored `created_at` is always `.sssZ`, so a raw whole-second bound text-compares `'…00.000Z' >= '…00Z'` = FALSE and silently drops the boundary row. JavaScriptCore rolls impossible calendar dates over (`2026-02-30` → Mar 2) → reject via a parse round-trip.
- **Host-CLI config seam (`collector/config.ts`):** `REVU_OWNER_MAP_FILE` = a JSON `coder.owner`→`{email,displayName?}` map, the in-gate stand-in for the real host `repos.map`/Coder API (M6 swaps the source behind the same `CoderOwnerResolver` seam); `REVU_HOST_DATA_DIR` = the host store dir. Config errors are content-free (never echo file bytes — a JSON-parse message can quote a mispasted secret).

**M4 (live layer + identity ground truth — poll loop, commentAuthors, BrokerPullMeta; PRs #43/#44/#45). New code under `packages/revud/src/broker/poll-loop.ts` + `reviewer-assignment.ts`:**
- **`authorHumanId` is a host-side COLLECTOR field (M6), NOT in-workspace-derivable.** `RevuApi` has NO PR-creation method — revu is a review client; contractors open PRs directly as the App (mint-and-inject). So which contractor drove an App-authored PR is NOT in `pull.user.login` (that's the bot) nor any in-workspace journal (reviewing ≠ authoring). M4.2 ships the `pr_author(pr PK, human_id NULL)` SEAM in the workspace store (`getPrAuthor`, first-write-wins via `ON CONFLICT(pr) DO NOTHING`); **the M6 collector populates it via the `coder.owner`↔PR correlation** the docker-exec tick has. Do NOT derive `authorHumanId` from the login→human map (backwards: that yields non-null for org members, whom the type wants `null`) or from write rows. `canApprove` is INDEPENDENT and derivable now: `pull.user.login.toLowerCase() !== botLogin.toLowerCase()` (App-authored ⟺ the bot login → `false`).
- **Broker-served meta MUST be reflected in the broker ETag, or a meta-only change never reaches the frontend.** The broker ETag derives from GitHub's list ETag (M4.1), but `BrokerPullMeta` also depends on state OUTSIDE the list (`reviewers.yaml`, `pr_author`). If a meta-only change doesn't move the ETag, the conditional client gets a 304 and keeps stale meta forever. Compose `brokerEtag = W/"pulls:<djb2(listEtag)>:<djb2(annotations)>"` where `annotations` = a deterministic serialization of `[{pr, authorHumanId, canApprove, assignedReviewerHumanIds}]` sorted by pr — meta scalars/arrays ONLY, never the GitHub pull objects (their float/field-order drift is exactly why the list ETag, not a body hash, anchors it). Recompute it on the 200 AND the 304 tick. Restart-stable because every input is durable. The mock oracle hashes items INCLUDING meta, so a meta change flips ITS ETag — the broker must match (else the frontend can tell mock from broker).
- **Poll-loop failure-counter ordering IS the stale-serving tripwire.** Reset the consecutive-failure counter ONLY after a FULLY-successful tick (list read AND facts phase resolved AND cache assigned) + in the warm-304 branch. Resetting right after the list read lets a "list-OK / facts-failing" state (GraphQL has a SEPARATE rate bucket from REST) oscillate the counter 0↔1 forever, never trip the stale→502, and serve an indefinitely stale list as fresh 200s. Isolate per-pull `getCompare` failures (carry that pull's prior meta forward, continue the tick); a batched-facts (GraphQL) failure DOES fail the whole tick. Cold cache (never warmed) or over-threshold → throw `PollUnavailableError` → 502 `broker_unreachable` (never empty-as-fresh). A 304 with a cold cache is a nonconforming upstream — treat as no-op, never fabricate an empty list. Keep `reviewers.load()` INSIDE the tick's try/finally (a throwing load must not wedge the in-flight flag).
- **Change-detection must include `base.sha`.** A base-branch advance under an unchanged head bumps neither `head.sha` nor `updated_at`, but moves the three-dot merge base → `compareKey`. Omit `base.sha` from the poll's change-detection and the frontend's "base moved" staleness (head unchanged + base advanced) can never fire in broker mode. (Pairs with the GitHub landmine that a base advance doesn't refresh the pull's stored `base.sha` until a synchronize — the list payload's `base.sha` is what to compare.)
- **`submitReview` journals the REVIEW id, not the inline COMMENT ids** — but `commentAuthors` is keyed by comment id. For exact own-comment detection on review inline comments (the primary comment type), journal each created inline comment id under a distinct `submitReviewComment` endpoint (read `GET /pulls/{n}/reviews/{id}/comments` after a confirmed submit). **ISOLATE that journaling (try/catch, non-fatal)** — it runs AFTER the review has landed on GitHub, so a journaling failure must never turn a confirmed submit into a user-visible failure (which keeps the draft → double-post on resubmit). Skip the extra GET for zero-inline-comment submits. `commentAuthors` assembly filters to comment-CREATING endpoints ONLY (`replyToThread` + `submitReviewComment`); reactions/resolves are not authorship. Read the WORKSPACE store's `listAudit`, NEVER the host store. In the out-of-band detector, `submitReviewComment` ids are DELIBERATELY touch-classified (absolved via the parent review's id) — never move them to the absolving set.
- **Don't hand-roll a YAML parser for lead-edited config — use `Bun.YAML.parse` (Bun ≥1.3).** A hand-rolled parser silently blanked last-good on a typo'd/miscapitalized/mis-indented section, coerced exotic keys onto the wrong PR, and corrupted lists/quoted scalars. Even with a real parser: validate the shape, STRICT integer PR keys (`/^\d+$/`, reject → keep last-good, never route to a wrong PR), keep last-good on any read/parse/validation failure (warn, NEVER echo file bytes — a YAML error can quote a mispasted secret), and distinguish a present-but-empty section (honor as an intentional clear) from an absent/unrecognized section in a non-empty file (keep last-good). `reviewers.yaml` = `humans:` (login→Human.id) + `assignments:` (pr→[Human.id]), from `REVU_REVIEWERS_FILE` (default `${REVU_DATA}/reviewers.yaml`), re-read per tick. `Bun.YAML` typing: the project avoids `@types/bun`, so add a minimal ambient `Bun.YAML.parse(text): unknown` to `packages/revud/src/bun-env.d.ts` (runtime built-in, revud-only, no bundle impact).
